# 0.8 Phase 3 Implementation Plan: SQL Generation on Query Plan

**Status:** Next.

**Created:** 2026-06-27.

## Purpose

Phase 3 moves SQL generation and SQL-facing diagnostics from Remotion clause types to the DataLinq-owned query plan introduced in Phase 2.

This is the phase where the plan stops being a snapshot artifact and becomes the production translation boundary. Remotion may still parse expression trees and still call `QueryExecutor` through `IQueryExecutor`, but SQL generation must consume `DataLinqQueryPlan`, `QueryPlanOperation`, `QueryPlanPredicate`, `QueryPlanValue`, source slots, bindings, projections, and result kinds.

The important distinction:

- acceptable after Phase 3: `QueryExecutor` receives a Remotion `QueryModel` at the outer parser seam, converts it to a DataLinq plan, and uses the plan for SQL
- not acceptable after Phase 3: SQL predicates, orderings, paging, scalar aggregate SQL, relation `EXISTS`, or join SQL are generated by visiting Remotion `WhereClause`, `OrderByClause`, `QuerySourceReferenceExpression`, `SubQueryExpression`, or result-operator objects

Phase 3 is successful when the current public query behavior still passes, but the main SQL path no longer cares that Remotion was the parser.

## Current Translation Boundary

The current production path is still Remotion-shaped:

- `src/DataLinq/Linq/QueryExecutor.cs`
  - accepts `QueryModel`
  - builds `SqlQuery<T>` through `BuildSqlQuery`
  - iterates Remotion body clauses and result operators
  - handles explicit join SQL by inspecting Remotion `JoinClause`
  - computes scalar aggregate selectors from Remotion `SelectClause`
- `src/DataLinq/Query/SqlQuery.cs`
  - exposes `Where(WhereClause)`
  - exposes `OrderBy(OrderByClause)`
- `src/DataLinq/Linq/Visitors/WhereVisitor.cs`
  - visits Remotion/Expression nodes and mutates `WhereGroup`
- `src/DataLinq/Linq/Visitors/OrderByVisitor.cs`
  - visits Remotion `Ordering`
- `src/DataLinq/Linq/QueryBuilder.cs`
  - contains valuable SQL details, but is tied to Remotion query-source expressions and mutable visitor state

The lower-level SQL builder is still useful:

- `SqlQuery<T>`
- `Select<T>`
- `WhereGroup<T>`
- `Where<T>`
- `Operand`
- `Join<T>`
- provider escaping and `DatabaseProvider.GetSqlForFunction(...)`

Do not throw that away in Phase 3. The target is not a greenfield SQL generator. The target is a DataLinq plan-to-`SqlQuery<T>` renderer that reuses the stable SQL primitives while removing Remotion from the renderer inputs.

## Non-Goals

- no DataLinq expression parser
- no replacement of Remotion `QueryableBase<T>` or `IQueryExecutor`
- no removal of the `Remotion.Linq` package
- no broad join expansion
- no subquery pushdown for post-paging operators
- no `GroupBy(...)`
- no broad nested database subqueries
- no non-SQL executor
- no AOT-clean projection interpreter
- no public query-plan API
- no deliberate LINQ behavior expansion

Projection execution can remain partly tied to the current expression/projection machinery until Phase 5. Phase 3 is about SQL generation and SQL diagnostics, not completing the projection interpreter.

## Design Rule

The renderer must be a query-plan consumer.

Good:

```text
Remotion QueryModel
  -> RemotionQueryPlanAdapter
  -> DataLinqQueryPlan
  -> QueryPlanSqlBuilder
  -> SqlQuery<T> / Select<T>
  -> provider SQL
```

Bad:

```text
Remotion QueryModel
  -> DataLinqQueryPlan
  -> QueryPlanSqlBuilder
  -> WhereVisitor / OrderByVisitor / QuerySourceReferenceExpression
  -> SqlQuery<T>
```

If a Phase 3 SQL helper needs `WhereClause`, `OrderByClause`, `Ordering`, `QuerySourceReferenceExpression`, `SubQueryExpression`, `IQuerySource`, or Remotion result operators, the migration is incomplete.

## Proposed Code Layout

Prefer keeping SQL rendering under the internal LINQ planning boundary:

```text
src/DataLinq/Linq/Planning/Sql/
  QueryPlanSqlBuilder.cs
  QueryPlanSqlPredicateBuilder.cs
  QueryPlanSqlValueRenderer.cs
  QueryPlanSqlSourceMap.cs
  QueryPlanSqlProjectionPlanner.cs
  QueryPlanSqlDiagnostics.cs
```

Rationale:

- `DataLinq.Linq.Planning` already owns the plan model.
- `DataLinq.Query` is the lower-level SQL builder and should not learn Remotion or plan parsing concerns.
- Keeping this internal prevents accidental public API lock-in.
- A nested `Sql` folder makes it obvious which classes are backend-specific consumers of the backend-neutral plan.

The new `Planning/Sql` files must not import Remotion namespaces.

## Workstream A: Renderer Skeleton and Source Map

Goal: create a plan-to-SQL skeleton with no behavior change until it is explicitly routed.

Tasks:

1. Add `QueryPlanSqlBuilder<T>`.
2. Add a source map that resolves `QueryPlanSourceSlot.Id` to:
   - table
   - alias
   - materialization role
   - selected primary-key aliases for join execution
3. Validate there is exactly one root table source.
4. Build the root `SqlQuery<T>` from the root source:
   - table from `QueryPlanSourceSlot.Table`
   - alias from `QueryPlanSourceSlot.Alias`
   - transaction/data source from the existing `QueryExecutor`
5. Keep operation application ordered.
6. Add early unsupported diagnostics for impossible plan shapes, not null reference failures.
7. Add unit tests for source-map validation if the logic is non-trivial.

Exit criteria:

- a simple root query plan can create a `SqlQuery<T>` with the correct table and alias
- the new renderer compiles without Remotion imports
- no production route changes yet

## Workstream B: Binding Frame and Value Rendering

Goal: render plan values from plan shape plus binding values.

Tasks:

1. Add `QueryPlanSqlValueRenderer`.
2. Resolve `QueryPlanCapturedValue` through `QueryPlanBindingFrame`.
3. Resolve `QueryPlanLocalSequenceValue` through `QueryPlanBindingFrame`.
4. Reject missing, duplicate, or wrong-kind bindings with `QueryTranslationException`.
5. Render `QueryPlanColumnValue` as an `Operand.Column(...)` with the correct source alias.
6. Render scalar captured values as `Operand.Value(...)`.
7. Render local sequences as value arrays for `IN` and `NOT IN`.
8. Normalize values against column types where the existing translator does today:
   - char comparisons
   - nullable wrapper values
   - provider writer conversion when needed
9. Keep SQL parameterization. Do not inline captured values into SQL text.
10. Add tests proving different captured scalar values produce the same SQL template shape when existing template caching can apply.

Design stance:

- SQL rendering may look at binding values to add parameters.
- SQL rendering must not put captured values into diagnostics, snapshots, or SQL text.
- Local sequence count can affect SQL shape. Local sequence element values must remain parameters.

Exit criteria:

- scalar comparisons bind through the frame
- local membership binds through the frame
- redaction behavior proven by existing plan snapshots remains intact

## Workstream C: Predicate Renderer

Goal: replace `WhereVisitor` and Remotion-backed predicate mutation with explicit predicate-node rendering.

Tasks:

1. Add `QueryPlanSqlPredicateBuilder<T>`.
2. Render `QueryPlanPredicate.Fixed` as `Operator.AlwaysTrue` or `Operator.AlwaysFalse`.
3. Render `And` and `Or` by creating explicit `WhereGroup<T>` subgroups.
4. Render `Not` through a negated subgroup or a directly negated leaf where that is equivalent.
5. Render `Compare` for:
   - column-to-value
   - value-to-column
   - column-to-column
   - function-to-value
   - value-to-function
6. Preserve comparison operator reversal when the column is on the right side.
7. Preserve SQL null handling:
   - `== null` -> `IS NULL`
   - `!= null` -> `IS NOT NULL`
   - nullable non-null inequality with `CSharpNullableNotEqualIncludesNull` -> `column <> value OR column IS NULL`
   - nullable bool inequality behavior already covered by compliance tests
8. Render `In` with `IN` or `NOT IN`.
9. Keep empty local sequences out of `In`; they should already be `Fixed`.
10. Render `Exists` through relation subquery SQL.
11. Keep diagnostic messages in DataLinq terms:
   - "query plan predicate"
   - "source slot"
   - "relation exists"
   - "local sequence binding"
12. Delete or bypass mutable visitor state for `ORs` and pending negations in the migrated path.

Implementation guidance:

- Reuse `WhereGroup<T>`, `Where<T>`, `Comparison`, and `Operand`.
- Move reusable value normalization from `QueryBuilder` into a Remotion-free helper if needed.
- Do not call `WhereVisitor.Parse(...)`.
- Do not call `QueryBuilder.GetOperand(Expression)` from the plan renderer.

Exit criteria:

- all supported single-source predicate shapes render from plan nodes
- fixed-condition truth tables still render `1=0` and `1=1`
- diagnostics for unsupported plan values do not mention Remotion classes

## Workstream D: Function Rendering

Goal: preserve provider-specific SQL function behavior from `QueryPlanFunctionKind`.

Tasks:

1. Map function-value nodes to SQL operands.
2. Preserve string predicate behavior:
   - `StartsWith` -> `LIKE value + "%"`
   - `EndsWith` -> `LIKE "%" + value`
   - string `Contains` -> `LIKE "%" + value + "%"`
3. Preserve string value functions:
   - `Length`
   - `Trim`
   - `ToUpper`
   - `ToLower`
   - `Substring`
4. Preserve static string checks:
   - `IsNullOrEmpty`
   - `IsNullOrWhiteSpace`
5. Preserve date/time member functions:
   - year
   - month
   - day
   - day-of-year
   - day-of-week
   - hour
   - minute
   - second
   - millisecond
6. Use provider function rendering through `DatabaseProvider.GetSqlForFunction(...)` for functions already modeled by `SqlFunctionType`.
7. Add explicit rendering logic for LIKE-based predicates because `SqlFunctionType` does not currently model `StartsWith`, `EndsWith`, or string `Contains`.
8. Add provider-matrix parity tests for string/date/time predicates touched by the new renderer.

Exit criteria:

- function SQL remains provider-specific where it was provider-specific before
- SQLite-only success is not treated as enough for function rendering changes

## Workstream E: Ordering, Paging, and Result SQL

Goal: apply plan operations and result kinds without Remotion result operators.

Tasks:

1. Apply `QueryPlanOperation.OrderBy` through source-slot-aware column/value rendering.
2. Apply `Skip` and `Take` from plan values.
3. Reject non-constant/non-scalar paging counts if they are not currently supported.
4. Preserve current result limits:
   - `Single` and `SingleOrDefault` use `LIMIT 2`
   - `First` and `FirstOrDefault` use `LIMIT 1`
   - scalar `Any` may use scalar count behavior, matching current execution
5. Preserve current `Last` and `LastOrDefault` behavior unless the existing test baseline says otherwise.
6. Keep the Phase 1/2 explicit post-paging rejection intact until subquery pushdown is deliberately implemented.
7. Keep operation order in the renderer even when current supported shapes flatten cleanly to one SQL query.
8. Add tests around:
   - `Where(a).Where(b)`
   - `Where(a).OrderBy(...).Where(b)`
   - `Where(a).Where(b).Skip(...).Take(...)`
   - explicit rejection of `Skip(...).Where(...)`

Exit criteria:

- ordering and paging no longer depend on `OrderByClause`, `Ordering`, `SkipResultOperator`, or `TakeResultOperator`
- current supported ordering/paging results remain equivalent

## Workstream F: Scalar Results and Aggregates

Goal: render scalar SQL from `QueryPlanResult`.

Tasks:

1. Render `Count` as `COUNT(*)`.
2. Render `Any` with the current scalar result behavior and conversion to `bool`.
3. Render `Sum`, `Min`, `Max`, and `Average` using `QueryPlanResult.AggregateSelector`.
4. Preserve current `Sum` null behavior:
   - direct numeric non-null `Sum` uses `COALESCE(SUM(column), 0)`
   - nullable aggregate conversion still returns default/null according to current tests
5. Validate aggregate selectors are direct numeric column values or supported nullable unwraps.
6. Reject computed aggregate selectors with DataLinq-owned diagnostics.
7. Move `GetAggregateSelectorSql(...)` logic off Remotion `SelectClause` and onto plan values.
8. Keep `ConvertScalarResult(...)` behavior unless tests prove it needs adjustment.

Exit criteria:

- scalar aggregates no longer inspect Remotion selectors
- aggregate support remains exactly the documented narrow boundary

## Workstream G: Relation EXISTS

Goal: render relation existence predicates from `QueryPlanPredicate.Exists`.

Tasks:

1. Use `RelationProperty` from the predicate node.
2. Validate the relation part is the supported candidate-key side.
3. Build a child `SqlQuery<object>` using `ChildSource.Table` and `ChildSource.Alias`.
4. Add parent-child key comparisons using source-slot aliases.
5. Render optional child predicates through the same plan predicate renderer.
6. Preserve nested child predicate grouping with `And` and `Or`.
7. Preserve nullable child comparison semantics.
8. Preserve negated `Any(...)` and existence-equivalent `Count()` behavior.
9. Add tests comparing old SQL/results to plan-rendered SQL/results for:
   - relation `Any()`
   - relation `Any(predicate)`
   - negated relation `Any(predicate)`
   - `Count() == 0`
   - `Count() > 0`

Exit criteria:

- relation `EXISTS` SQL no longer depends on Remotion `SubQueryExpression`, `MainFromClause`, or relation predicate expression visitors
- unsupported relation predicate diagnostics still name the relation and supported boundary

## Workstream H: Explicit Join Baseline

Goal: migrate the current narrow explicit inner join SQL to source slots.

Tasks:

1. Render `QueryPlanOperation.Join` into `SqlQuery<T>.Join(...)`.
2. Use source-slot aliases from the plan.
3. Render only `QueryPlanJoinKind.Inner`.
4. Add join equality with direct column operands from `QueryPlanJoin.LeftColumn` and `RightColumn`.
5. Select primary keys for every joined source using deterministic aliases.
6. Materialize rows by source slot instead of Remotion query-source identity.
7. Keep row-local projection execution on the existing projection evaluator until Phase 5 if necessary.
8. Preserve unsupported-shape diagnostics from the adapter:
   - `GroupJoin`
   - composite keys
   - filters/orderings/paging over joined rows
   - relation-property projection in join selector

Design stance:

- Join SQL generation should be plan-owned in Phase 3.
- Join projection evaluation can still use transitional Remotion-derived projection binding if the plan projection is not yet rich enough to execute anonymous projections directly.
- Do not expand join support while moving the boundary. Phase 8 owns broader join work.

Exit criteria:

- current narrow join tests pass through the plan SQL path
- join SQL rendering no longer inspects `JoinClause`

## Workstream I: Projection and Materialization Boundary

Goal: keep current projection behavior while making the SQL query source plan-owned.

Tasks:

1. Split SQL planning from projection execution in `QueryExecutor`.
2. Use the plan to decide what SQL to select:
   - full entity rows for entity and post-materialization projections
   - scalar aggregate selectors for aggregate scalar queries
   - primary-key selectors for explicit joins
3. Continue using the existing projection evaluator for row-local projections where necessary.
4. Keep relation-property projection rejection intact.
5. Avoid claiming projection is AOT-clean in Phase 3.
6. Leave clear TODO markers pointing to Phase 5 only where they clarify temporary Remotion projection dependencies.
7. Add guard tests that SQL generation passes without Remotion even if projection execution still needs the existing selector expression.

Exit criteria:

- SQL generation for projection queries is plan-owned
- current scalar, anonymous, and computed row-local projection tests pass
- temporary projection dependencies are isolated and easy to remove later

## Workstream J: QueryExecutor Routing

Goal: make production execution use the plan SQL renderer.

Tasks:

1. Add a private `CreateQueryPlan(...)` seam in `QueryExecutor`.
2. Convert incoming `QueryModel` using `RemotionQueryPlanAdapter`.
3. Replace `BuildSqlQuery<T>(QueryModel)` with `BuildSqlQuery<T>(DataLinqQueryPlan)`.
4. Replace `ParseQueryModel<T>(QueryModel)` with a plan-backed implementation.
5. Route collection execution through the plan-backed SQL query.
6. Route scalar execution through the plan-backed SQL query.
7. Route explicit join execution through plan-backed SQL query construction.
8. Keep any remaining `QueryModel` use limited to:
   - outer parser compatibility
   - current projection expression evaluation until Phase 5
   - transitional error context where no DataLinq-owned context exists yet
9. Remove production calls to:
   - `SqlQuery<T>.Where(WhereClause)`
   - `SqlQuery<T>.OrderBy(OrderByClause)`
   - `WhereVisitor.Parse(...)`
   - `OrderByVisitor.Parse(...)`
10. Prefer deleting unused Remotion visitor code only if the compiler and migration sequence make it safe. Otherwise leave it as clearly temporary dead code for Phase 7 cleanup.

Exit criteria:

- production query execution creates SQL through `DataLinqQueryPlan`
- Remotion remains a producer, not the SQL semantic model

## Workstream K: Compatibility Methods and Cleanup

Goal: make it hard for the old Remotion-shaped SQL boundary to come back accidentally.

Tasks:

1. Search for all callers of:
   - `SqlQuery<T>.Where(WhereClause)`
   - `SqlQuery<T>.OrderBy(OrderByClause)`
   - `WhereVisitor`
   - `OrderByVisitor`
   - `QueryBuilder`
2. Remove or narrow calls from active production paths.
3. If public Remotion-shaped `SqlQuery<T>` methods cannot be removed yet, mark them with migration comments and tests proving production does not call them.
4. Add an architectural test that the new `DataLinq.Linq.Planning.Sql` namespace has no Remotion references.
5. Add an architectural test or focused reflection assertion that plan node and plan SQL renderer types do not expose Remotion types in public/internal member signatures.
6. Keep `RemotionQueryPlanAdapter` as the only expected Remotion dependency inside `DataLinq.Linq.Planning`.

Exit criteria:

- active SQL generation path has no Remotion-shaped API dependency
- remaining Remotion-shaped APIs are documented as temporary cleanup targets

## Workstream L: SQL Parity Tests

Goal: prove the new renderer is equivalent before and after routing production.

Tasks:

1. Add a test helper that can build SQL through:
   - the phase-start Remotion/visitor path before routing
   - the new plan renderer
2. Normalize SQL enough to compare meaningfully without hiding provider differences.
3. Compare SQL text and parameter counts for representative SQLite shapes where stable.
4. Prefer result parity over exact SQL text for provider-specific function behavior.
5. Cover:
   - chained `Where`
   - `Where`/`OrderBy`/`Where`
   - fixed true/false conditions
   - local `Contains`
   - local `Any(predicate)` membership
   - nullable non-null inequality
   - string/date/time functions
   - relation `EXISTS`
   - relation `Count()` existence comparisons
   - scalar aggregates
   - post-materialization projection
   - explicit join
6. Once production routes through the plan path, keep old-vs-new SQL parity tests only as migration scaffolding. They should not block Phase 7 cleanup by permanently rooting the old visitors.

Exit criteria:

- changed SQL generation has focused parity coverage
- tests catch dropped predicates and dropped ordering

## Workstream M: Diagnostics

Goal: make SQL-generation failures use DataLinq concepts.

Tasks:

1. Replace result-operator type-name diagnostics in migrated paths with `QueryPlanResultKind`.
2. Replace body-clause diagnostics in migrated paths with `QueryPlanOperationKind`.
3. Include source slot ids and table names when a column cannot be resolved.
4. Include relation property names for relation `EXISTS` failures.
5. Include function kind names for unsupported function rendering.
6. Avoid raw Remotion `QueryModel` strings in new renderer diagnostics.
7. Keep messages actionable:
   - what shape failed
   - what is supported
   - where to move the LINQ operation if the current limitation is operator order

Exit criteria:

- unsupported-shape compliance tests still pass
- new renderer failures do not leak Remotion class names

## Recommended Implementation Order

1. Add the plan SQL renderer skeleton and source map.
2. Add value/binding rendering.
3. Add predicate rendering for fixed, comparison, boolean groups, and local membership.
4. Add ordering, skip, take, and result limits.
5. Add scalar aggregate rendering.
6. Add relation `EXISTS`.
7. Add explicit join rendering.
8. Add SQL parity tests while the old path still exists.
9. Route `QueryExecutor` collection and scalar paths through the plan renderer.
10. Route explicit join execution through the plan renderer.
11. Remove or isolate old production calls to Remotion-shaped SQL helpers.
12. Run provider matrix verification.

This order keeps the old path available as an oracle until the plan renderer is ready. Removing the old path first would be theatrical, not useful.

## Risk Register

| Risk | Severity | Mitigation |
| --- | --- | --- |
| The plan renderer accidentally calls Remotion helpers | High | Add namespace/signature guard tests and code review rule for `Planning/Sql`. |
| SQL changes look equivalent on SQLite but fail on MySQL/MariaDB | High | Run provider-backed lanes for function, nullable, aggregate, and relation changes. |
| Boolean grouping regresses during visitor-state removal | High | Add focused predicate renderer tests plus existing boolean logic compliance tests. |
| Captured values leak into SQL text or diagnostics | Medium | Bind through `QueryPlanBindingFrame`; assert SQL parameterization and redacted snapshots. |
| Nullable inequality loses C# lifted semantics | High | Keep explicit `CSharpNullableNotEqualIncludesNull` rendering and tests. |
| Projection migration scope creeps into Phase 3 | Medium | Keep projection execution isolated and defer AOT-clean interpreter work to Phase 5. |
| Join migration expands support accidentally | Medium | Render only the Phase 1/2 narrow join baseline; leave broader joins to Phase 8. |
| Old visitor code remains and confuses future work | Medium | Stop production callers now; delete or clearly mark cleanup targets for Phase 7. |

## Exit Criteria

Phase 3 is complete when:

- production SQL generation consumes `DataLinqQueryPlan`
- `SqlQuery<T>.Where(WhereClause)` and `SqlQuery<T>.OrderBy(OrderByClause)` are no longer the main translation boundary
- the new plan SQL renderer has no Remotion imports
- supported single-source predicates, ordering, paging, fixed conditions, local membership, nullable predicates, functions, scalar aggregates, relation `EXISTS`, projections, and the narrow explicit join baseline still pass
- unsupported post-paging operators remain explicitly rejected
- unsupported query diagnostics in migrated SQL paths use DataLinq plan concepts rather than Remotion class names
- provider compliance tests pass for the changed SQL generation surface
- remaining Remotion use is limited to parser production and temporary projection compatibility seams that later phases explicitly own

## Phase 4 Handoff

Phase 4 should start only after Phase 3 proves that SQL generation can consume a DataLinq plan without caring who produced it.

The handoff should include:

- the stable plan node set actually required by SQL generation
- plan SQL renderer tests
- any plan-model adjustments discovered during rendering
- list of remaining temporary Remotion dependencies in projection or test scaffolding
- unsupported-shape diagnostics that the DataLinq expression parser must preserve

The Phase 4 parser should then target the same plan. It should not invent a second query model.
