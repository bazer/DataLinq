# 0.8 Query Parser Overview

**Status:** Consolidation source for the version-scoped [DataLinq 0.8 Roadmap](../v0.8/README.md).

**Created:** 2026-06-27.

## Purpose

0.8 should make the Remotion replacement work explicit instead of leaving it scattered across roadmap, AOT, query-runtime, and archived implementation notes.

The short version: replacing `Remotion.Linq` is the right 0.8 theme, but deleting it first would be reckless. The safe path is to make Remotion one temporary parser feeding a DataLinq-owned query plan, move SQL generation behind that plan, then build the DataLinq expression parser against the same plan and prove parity before flipping defaults.

The desired 0.8 product outcome is:

- DataLinq owns the query plan and diagnostics boundary.
- The supported generated/AOT query path no longer roots `Remotion.Linq`.
- The documented LINQ support matrix remains true, or any narrower 0.8 parser boundary is documented honestly.
- `Remotion.Linq` is either removed from the main runtime package or isolated as an explicit compatibility path outside the practical AOT support boundary.

If 0.8 only swaps parser code and quietly regresses supported LINQ shapes, that is not progress. It is a rewrite wearing a release number.

## Current Dependency Shape

`Remotion.Linq` is not a neat package reference we can cut out with a weekend patch.

Current runtime and test touchpoints include:

- `src/DataLinq/DataLinq.csproj`, which references `Remotion.Linq`.
- `src/Directory.Packages.props`, which pins `Remotion.Linq` 2.2.0.
- `src/DataLinq.AotSmoke` and `src/DataLinq.TrimSmoke`, which currently root `Remotion.Linq` for constrained publish analysis.
- `src/DataLinq/Linq/Queryable.cs`, which inherits Remotion `QueryableBase<T>` and creates `QueryParser.CreateDefault()`.
- `src/DataLinq/Linq/QueryExecutor.cs`, which consumes `QueryModel`, body clauses, result operators, selectors, and joined query models.
- `src/DataLinq/Linq/QueryBuilder.cs`, which understands Remotion query-source references and subquery expressions for predicates, local collections, relation `EXISTS`, and aggregate-adjacent shapes.
- `src/DataLinq/Linq/Visitors/WhereVisitor.cs` and `OrderByVisitor.cs`, which accept Remotion clause types directly.
- `src/DataLinq/Linq/LocalSequenceExtractor.cs`, `ProjectionExpressionEvaluator.cs`, and `Evaluator.cs`, which special-case Remotion expression nodes.
- `src/DataLinq/Query/SqlQuery.cs`, whose public internal translation methods still accept Remotion `WhereClause` and `OrderByClause`.
- compliance tests that instantiate Remotion's parser directly or reflect into `ParseQueryModel`.

That means the real migration boundary is the query intermediate representation. The package reference disappears near the end, not at the beginning.

## 0.8 Design Center

### DataLinq Query Plan

The plan must represent query intent, not SQL text and not Remotion clause trivia.

Minimum plan concepts:

- source slots with stable aliases, table metadata, element types, and nullability/cardinality flags for future joins
- an ordered query operation pipeline or nested plan nodes so LINQ operator order is preserved
- predicate nodes for `And`, `Or`, `Not`, comparison, membership, fixed true/false, and relation existence
- value nodes for columns, constants, captured values, local sequences, converted provider values, and provider functions
- ordering and paging nodes that can express subquery pushdown when needed
- projection nodes for entity, scalar member, anonymous projection, and carefully bounded computed row-local projection
- result operators for sequence, single/first/last variants, count, any, and scalar aggregates
- a binding frame that separates query shape from captured runtime values

The operator-order point is not academic. These are different queries:

```csharp
source.OrderBy(x => x.Name).Take(10)
source.Take(10).OrderBy(x => x.Name)
source.OrderBy(x => x.HireDate).Take(10).OrderBy(x => x.Name)
```

The first can usually render as one flat SQL query. The second and third need subquery pushdown to preserve LINQ semantics. Flattening them into final `ORDER BY`/`LIMIT` SQL clause order would be wrong.

### Remotion Adapter First

The first producer should be `RemotionQueryPlanAdapter`.

That sounds backwards until you look at the blast radius. It gives us:

- current behavior preserved while SQL generation moves behind DataLinq nodes
- plan snapshot tests before parser replacement
- a direct parity target for the new expression parser
- one SQL generator path instead of parallel Remotion and DataLinq translator stacks

The adapter is scaffolding. It should not become the permanent architecture.

### DataLinq Expression Parser Second

The new parser should parse `System.Linq.Expressions` directly into `DataLinqQueryPlan`.

First parser slice should target the documented support matrix, not arbitrary LINQ:

- `Where`
- `OrderBy`, `OrderByDescending`, `ThenBy`, `ThenByDescending`
- `Select`
- `Skip` and `Take`
- `Any` and `Count`
- `Single`, `SingleOrDefault`, `First`, `FirstOrDefault`, `Last`, `LastOrDefault`
- documented scalar aggregates
- local `Contains(...)`, empty fixed-condition behavior, and equality-shaped local `Any(predicate)`
- documented string/date/time/nullable predicate shapes
- generated one-to-many relation `Any(...)` and existence-equivalent `Count()` predicates if feasible in 0.8
- the current narrow explicit `Join(...)` baseline if feasible in 0.8

If relation predicates or the current join baseline cannot make the first parser slice, the release must say that clearly and keep the compatibility path explicit. Hiding a narrower parser behind the old public support wording would be dishonest.

## Proposed 0.8 Work Packages

### Package 0: Baseline And Test Contract

Goal: lock down what must not regress.

Tasks:

1. Treat the LINQ support matrix as the migration checklist.
2. Add missing SQL-shape or plan-shape assertions for high-risk cases:
   - chained `Where`
   - empty local collection fixed conditions
   - local projected `Contains`
   - equality-shaped local `Any(predicate)`
   - nullable comparison truth tables
   - relation `EXISTS`
   - scalar aggregates
   - row-local projections
   - the current narrow join baseline
   - `Take` or `Skip` followed by later `OrderBy`
3. Classify untested behavior as unsupported, not "probably fine".

Exit: the current Remotion path has an executable parity baseline.

### Package 1: Query Plan Model And Remotion Adapter

Goal: introduce the DataLinq-owned plan without changing public behavior.

Tasks:

1. Add immutable plan nodes and binding-frame types.
2. Add `RemotionQueryPlanAdapter`.
3. Add normalized plan snapshots for representative queries.
4. Keep the old SQL execution path available while the plan proves itself.

Exit: Remotion can parse a query and produce a DataLinq plan that tests can inspect.

### Package 2: SQL Generation Behind The Plan

Goal: make SQL generation consume DataLinq nodes instead of Remotion clauses.

Tasks:

1. Move predicate and ordering translation out of Remotion-specific visitors.
2. Resolve columns through source slots instead of visitor-global assumptions.
3. Preserve fixed true/false, local sequence, nullable, scalar aggregate, relation `EXISTS`, and current join semantics.
4. Replace `SqlQuery<T>.Where(WhereClause)` and `OrderBy(OrderByClause)` as the main boundary.
5. Add provider compliance coverage for SQLite, MySQL, and MariaDB where SQL generation changes.

Exit: Remotion remains only a parser producer; SQL generation and diagnostics consume DataLinq plan nodes.

### Package 3: DataLinq Query Provider And Parser

Goal: create the supported-subset parser over expression trees.

Tasks:

1. Add a DataLinq-owned `IQueryable<T>` and `IQueryProvider` path.
2. Parse method-call chains into plan operations.
3. Parse lambdas into predicate, ordering, projection, aggregate, and result nodes.
4. Rebuild local sequence extraction without Remotion `QueryModel`.
5. Add specific `QueryTranslationException` diagnostics for unsupported operators, expression shapes, and provider capability gaps.

Exit: the new parser passes the documented single-source subset and can run representative generated SQLite queries.

### Package 4: Projection And AOT Cleanup

Goal: keep generated/AOT execution honest.

Tasks:

1. Inventory reflection invocation in projection and local evaluation.
2. Replace row-member reads with generated metadata/accessor paths where practical.
3. Add a small interpreter or generated projector strategy for supported row-local projection shapes.
4. Keep compatibility fallback paths clearly outside the generated/AOT support boundary.

Exit: supported generated/AOT projection execution avoids reflection invocation and dynamic code in the hot path.

### Package 5: Dual-Run Parity And Switch

Goal: flip the supported path only after boring evidence.

Tasks:

1. Add a test mode that parses with Remotion and with the DataLinq parser.
2. Compare normalized plans.
3. Compare SQL templates where provider SQL is expected to remain equivalent.
4. Compare results across SQLite and available MySQL/MariaDB lanes.
5. Record intentional differences in the support matrix or dev plan.
6. Route generated SQLite AOT, trimmed, and WASM AOT smokes through the DataLinq parser.

Exit: the supported generated/AOT path does not root `Remotion.Linq`.

### Package 6: Remove Or Isolate Remotion

Goal: finish the dependency story.

Tasks:

1. Decide whether Remotion is deleted or moved to a compatibility package/mode.
2. Remove `Remotion.Linq` from `src/DataLinq/DataLinq.csproj` for the main runtime package if parity is sufficient.
3. Remove the central package version if no compatibility package remains.
4. Remove Remotion trimmer roots from smoke projects.
5. Verify package dependency groups and constrained publish outputs.

Exit: `Remotion.Linq` is absent from the main runtime package, or isolated outside the practical AOT support boundary with clear naming.

## Join And Relation Decision

Older roadmap order put Phase 13 and Phase 14 before Remotion isolation. Pulling Remotion replacement into 0.8 changes that.

My recommendation: do not implement broad multi-join or relation-aware join APIs on the old Remotion boundary first. That would increase the surface area we must migrate immediately afterward.

For 0.8:

- The plan model should be source-slot-aware from day one.
- The current narrow explicit `Join(...)` support should be preserved if practical.
- Current one-to-many relation `Any(...)` and existence-equivalent `Count()` predicates should be preserved if practical.
- Multi-join composition and relation-aware `JoinBy(...)`/left-join APIs should become follow-up work unless the parser migration naturally unlocks them without broadening scope.

Bluntly: a source-slot-aware query plan is foundational. A shiny join API before that foundation is churn.

## Non-Goals For 0.8

- general-purpose LINQ provider behavior
- `GroupBy(...)`
- broad nested database subqueries
- silent client-side predicate fallback
- SQL-backed projection expansion beyond the explicitly supported shapes
- a non-SQL backend executor as a release requirement
- DataLinq.Store query/module execution
- no-AOT browser WebAssembly support
- OPFS/file-backed browser storage
- warning suppression as the final answer for Remotion

## Release Gates

Minimum gates for claiming the 0.8 parser work is real:

- support-matrix tests pass for the enabled parser subset
- unsupported query diagnostics remain specific
- plan snapshot tests cover representative supported shapes
- dual-run parity is green for the shapes that both parsers support
- SQLite compliance quick suite passes
- MySQL/MariaDB lanes pass for SQL-generation changes when available
- Native AOT smoke publishes and runs without Remotion warnings in the supported path
- trimmed smoke publishes and runs without Remotion warnings in the supported path
- Blazor WASM AOT publish/browser smoke still passes if the release keeps that path in scope
- package inspection confirms the main runtime dependency story

Recommended local verification commands remain:

```powershell
.\scripts\dotnet-sandbox.ps1 run --project src\DataLinq.Testing.CLI -- run --suite compliance --alias quick --output failures --build
.\scripts\dotnet-sandbox.ps1 run --project src\DataLinq.Testing.CLI -- run --suite mysql --alias latest --output failures
.\scripts\dotnet-sandbox.ps1 publish src\DataLinq.AotSmoke\DataLinq.AotSmoke.csproj -f net10.0 -r win-x64 -c Release -v:minimal --self-contained true -p:PublishAot=true
.\scripts\dotnet-sandbox.ps1 publish src\DataLinq.TrimSmoke\DataLinq.TrimSmoke.csproj -f net10.0 -r win-x64 -c Release -v:minimal --self-contained true -p:PublishTrimmed=true
.\scripts\dotnet-sandbox.ps1 publish src\DataLinq.BlazorWasm\DataLinq.BlazorWasm.csproj -f net10.0 -c Release -v:minimal -p:RunAOTCompilation=true
```

As usual, WebAssembly build failures inside the native Windows Codex sandbox should be verified outside the sandbox before being classified as product bugs.

## Suggested First Slice

Start with a slice that proves the architecture without pretending the hard part is done:

1. Add a query-plan namespace and immutable plan node skeleton.
2. Add a Remotion-to-plan adapter for basic single-source `Where`, `OrderBy`, `Skip`, `Take`, and entity projection.
3. Add normalized plan snapshot tests.
4. Do not replace SQL generation yet.
5. Add one operator-order test for `Take(...).OrderBy(...)` or `Skip(...).OrderBy(...)` even if it initially documents current behavior as wrong or unsupported.

That slice is small enough to review and valuable enough to expose whether the plan shape is right. Starting by replacing `Queryable<T>` would maximize blast radius before the plan has earned trust.

## Open Decisions

1. Should 0.8 require complete removal of `Remotion.Linq` from the main package, or is a generated/AOT parser switch with explicit compatibility isolation enough?
2. Is current relation `Any(...)` parity mandatory for the first DataLinq parser, or can it remain compatibility-only until a second parser slice?
3. Is the current narrow explicit join baseline mandatory for the first DataLinq parser?
4. Should anonymous projection use an interpreter, generated projectors, or a compatibility-only fallback for the first AOT-clean path?
5. Should SQLitePCLRaw WASM warning disposition be included in 0.8, or deferred so the release stays centered on the query parser?

My opinion: answer 1 after the dual-run evidence exists, require relation and current join parity if it is not wildly expensive, prefer an interpreter before generated projectors, and defer SQLitePCLRaw warning disposition unless the WASM smoke becomes a release blocker.
