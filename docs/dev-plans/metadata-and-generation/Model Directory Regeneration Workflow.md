# Specification: Model Directory Regeneration Workflow

**Status:** Draft implementation plan.
**Goal:** Replace the separate `SourceDirectories`/`DestinationDirectory` generation workflow with one clearer `ModelDirectory` workflow. `create-models` should read existing model files from the model directory when preserving supported C# surface choices, write refreshed files back to that same directory, and offer `--fresh` for database-only regeneration.

## Executive Position

The separate source/destination model-tree idea should go. The useful behavior is not "read a second source model directory." The useful behavior is "regenerate models in place without destroying stable C# choices that users are allowed to make."

The cleaner contract is:

```text
ModelDirectory is the model output directory.
create-models reads existing files there when they exist.
create-models preserves supported C# surface choices from those files.
create-models writes refreshed generated files back there.
```

That is intuitive. Users already understand tools that update generated files in place while preserving specific stable edits. They do not understand why one model tree is called source, another is called destination, and both somehow participate in generation.

The current generated-file warning also needs nuance. A flat "do not edit this file" message is too blunt if DataLinq intentionally supports editing model class names, property names, relation names, and C# property types in generated model declarations. The honest rule is:

- It is OK to edit the supported model declaration surface that DataLinq reads back during regeneration.
- It is not OK to put arbitrary handwritten behavior in generated files and expect it to survive.
- Durable behavior belongs in partial classes, extension methods, or ordinary application code.
- Durable generation policy belongs in `datalinq.json`.

That distinction should be documented clearly and linked from the generated header.

## Current Code Audit

### Config Fields

`src/DataLinq/Config/ConfigFile.cs` currently exposes:

```csharp
public List<string>? SourceDirectories { get; set; }
public string? DestinationDirectory { get; set; }
```

`src/DataLinq/Config/DataLinqConfig.cs` lowers this into:

```csharp
public List<string> SourceDirectories { get; private set; }
public string? DestinationDirectory { get; private set; }
```

`datalinq.user.json` merges are database-name based. Scalar config values replace existing effective values when supplied.

### `create-models`

`src/DataLinq.CLI/Program.cs` defines:

```text
--skip-source
--overwrite-types
```

and maps `--skip-source` to:

```csharp
ReadSourceModels = !options.SkipSource
```

`src/DataLinq.Tools/ModelGenerator.cs` then:

1. reads provider metadata from the live database
2. optionally reads existing metadata from `SourceDirectories`
3. uses `MetadataTransformer` to merge source metadata into database metadata
4. renders files into `DestinationDirectory`

This workflow is the awkward part. It treats a second source tree as the preservation source and the generated output directory as the write target.

### Validation and Other CLI Paths

`src/DataLinq.Tools/SchemaValidator.cs` currently reads both `SourceDirectories` and `DestinationDirectory` when building model metadata for validation.

`src/DataLinq.Tools/ModelReader.cs` also starts from `SourceDirectories`, then includes `DestinationDirectory`.

`src/DataLinq.Tools/SqlGenerator.cs` and `src/DataLinq.Tools/DatabaseCreator.cs` already read model files from `DestinationDirectory` only.

This inconsistency is a symptom of the same problem. "Source directories plus destination directory" is not a crisp product concept.

### MetadataTransformer

`src/DataLinq.SharedCore/Factories/MetadataTransformer.cs` is valuable, but its current role is muddier than it needs to be.

The useful transformation is:

- database/provider metadata remains the schema source of truth
- existing model files provide supported C# surface choices
- transformed metadata keeps the database shape but preserves those choices

The transformer currently preserves several useful things from source metadata:

- database/model C# names and namespaces
- table collection property names
- scalar property names
- relation property names
- selected relation nullability
- model interfaces
- usings
- custom C# property types
- external enum references
- source enum default expressions
- some type/database type attributes
- optionally source or destination constraint names

Those are real features. This plan should keep the supported subset, but read it from the existing model directory rather than from a separate source directory.

## Desired Config Contract

### New Preferred Field

Add:

```json
"ModelDirectory": "Models"
```

to each database entry.

This should become the preferred field in docs and examples.

### Compatibility Alias

Keep:

```json
"DestinationDirectory": "Models"
```

as an alias for at least one compatibility window.

Recommended effective resolution:

1. If only `ModelDirectory` is set, use it.
2. If only `DestinationDirectory` is set, use it and emit a deprecation warning.
3. If both are set to the same normalized path, accept it.
4. If both are set to different paths, fail loudly.

Silently choosing one would create terrible bugs. A user who sets both differently is telling us the config is ambiguous.

### Removed Field

Deprecate and then remove:

```json
"SourceDirectories": [ "Models/Source" ]
```

Recommended transition behavior:

1. First implementation: still parse `SourceDirectories` so old config files do not explode during config deserialization.
2. `create-models` should ignore `SourceDirectories` and warn that it now reads existing files from `ModelDirectory`.
3. `validate` and `diff` should stop reading `SourceDirectories`.
4. A later breaking cleanup can remove the property from public config docs and eventually from the config model.

If we want a stricter first implementation, fail when `SourceDirectories` is present and non-empty. That is cleaner, but less friendly. My recommendation is a warning first, because the field appears in current docs and test configs.

## Desired CLI Contract

### `--fresh`

Add:

```text
--fresh
```

to `datalinq generate models` after the command-surface redesign.

Meaning:

```text
Ignore existing files in ModelDirectory and generate from database metadata plus datalinq.json only.
```

This is the right name because it describes the outcome without exposing the old implementation concept.

### `--skip-source`

Do not keep `--skip-source` as a deprecated alias in Phase 12C.

Earlier drafts considered a compatibility alias:

```text
Warning: --skip-source is deprecated. Use --fresh. create-models now reads existing models from ModelDirectory, not SourceDirectories.
```

The final command-surface decision is stricter: DataLinq is still pre-1.0, `SourceDirectories` is being removed from the generation workflow, and `--skip-source` becomes actively misleading once the command reads existing models from `ModelDirectory`. Users should get the parser's normal unknown-option error and the help text should show `--fresh`.

### `--overwrite-types`

Keep:

```text
--overwrite-types
```

This flag still has a reason to exist when `create-models` reads existing files from `ModelDirectory`. It answers:

```text
If an existing model property has a C# type different from the database-inferred type, should the database-inferred type replace it?
```

Default behavior should continue to preserve custom/source C# types where the current transformer considers that safe. `--overwrite-types` should continue to force simple inferred database types while preserving enum/custom-type cases that cannot be inferred reliably.

## Regeneration Workflow

Recommended `create-models` flow:

1. Resolve `ModelDirectory`.
2. Read provider metadata from the selected database connection.
3. Unless `--fresh` is set, check whether `ModelDirectory` exists and contains model files.
4. If existing model files are present, parse them as previous model output.
5. Transform the fresh provider metadata with supported C# surface choices from the previous model output.
6. Render every generated output file into memory.
7. If rendering succeeds, write refreshed files to `ModelDirectory`.

Important details:

- Missing `ModelDirectory` should not block first-time `create-models`; the command can create it before writing.
- Existing unreadable or invalid model files should be a real error unless `--fresh` is set.
- If `--fresh` is set, invalid existing model files should not matter because they are intentionally ignored.
- The transformer should be described as preserving supported model surface choices, not as merging a source model tree.

## Supported Edit Contract

This is the core user-facing contract.

When existing model files are read from `ModelDirectory`, DataLinq should preserve supported edits to the generated declaration surface.

Supported edits should include:

- database class C# name when matched to the configured database
- table/view model class C# names when matched by table/view database name
- table collection property names on the database model
- scalar property C# names when matched by `[Column(...)]`
- scalar property C# types when safe under the existing `--overwrite-types` semantics
- scalar property nullability where it represents a supported C# surface choice
- relation property names when matched by relation columns
- relation property nullability where it represents a supported C# surface choice
- enum declarations generated in model files
- references to external enum/custom types
- namespaces and usings needed by supported custom names/types

Unsupported or unsafe edits should include:

- arbitrary methods, fields, constructors, nested helper types, or business logic in generated files
- edits to generated implementation files from the source generator
- changing database mapping attributes casually, such as `[Column]`, `[Table]`, `[ForeignKey]`, or `[Relation]`, unless the user intentionally wants to change metadata
- edits to generated method bodies or helper sections that are not part of the abstract model declaration surface

This is not a moral rule. It is a mechanical rule. If DataLinq does not parse and preserve a shape deliberately, regeneration can overwrite it.

### Names and Types Are OK

The documentation should say plainly:

```text
It is OK to rename generated model classes and generated properties. DataLinq matches them back to database tables and columns using mapping attributes and preserves those C# names on the next regeneration.

It is also OK to use a custom C# property type, including shared enums, when DataLinq can preserve it safely. Use --overwrite-types when you intentionally want database-inferred types to replace existing simple C# types.
```

That is much clearer than pretending generated files are never edited.

### Mapping Attributes Need Care

The docs should also say:

```text
Do not edit [Column], [Table], [ForeignKey], or [Relation] attributes just to rename C# members. Those attributes describe database metadata. Change the C# class/property name instead.
```

This prevents the common footgun where a user changes `[Column("employee_id")]` to `[Column("EmployeeId")]` because they think it is a display name.

## Generated Header Policy

The existing generated-header plan says the stable header should be:

```csharp
// <auto-generated />
// This file was generated by DataLinq. Do not edit this file directly.
```

That wording should be revised when this workflow lands.

Recommended short header:

```csharp
// <auto-generated />
// Generated by DataLinq. Supported model names and C# property types may be edited and preserved on regeneration.
// Put behavior in partials. See: https://datalinq.dev/docs/model-generation/generated-models
```

If the final public docs URL is different, use the actual stable permalink. Do not put a local filesystem path or repo-relative markdown path in generated user files.

Rules:

- Keep the header short.
- Link to a real docs page for nuance.
- Do not list every supported edit in every generated file.
- Do not imply arbitrary edits are safe.
- Keep `<auto-generated />` first.

The source generator output should still use a stricter header, because compiler-generated implementation files are not the user-editable model declaration surface:

```csharp
// <auto-generated />
// Generated by DataLinq. Do not edit this compiler-generated implementation file.
```

## Documentation Plan

Add or update a dedicated docs page, for example:

```text
docs/model-generation/Generated Model Files.md
```

The page should answer practical questions directly:

### What DataLinq Generates

Explain that `create-models` writes abstract model declaration files into `ModelDirectory`, and the source generator creates implementation files during compilation.

### What You Can Edit

State that it is OK to edit:

- model class names
- database table collection property names
- scalar property names
- relation property names
- supported C# property types, including shared enums
- namespaces/usings required for those types

### What Not To Edit

State that arbitrary business logic, helper methods, validation methods, and computed properties should go into partial classes or extension methods, not generated files.

State that mapping attributes describe database metadata and should not be edited just to rename C# members.

### How Regeneration Works

Explain:

```text
datalinq create-models reads existing files from ModelDirectory, preserves supported model-surface choices, and writes refreshed files back to ModelDirectory.
```

Explain `--fresh`:

```text
Use --fresh when you want to ignore existing model files and recreate the model surface from database metadata and datalinq.json.
```

### Where Custom Code Goes

Show a tiny partial-class example:

```csharp
public abstract partial class Employee
{
    public string DisplayName => $"{FirstName} {LastName}";
}
```

Use an example that does not fight the generated abstract property pattern.

### How To Use Custom Types

Show a shared enum example:

```csharp
public enum OrderStatus
{
    Draft,
    Submitted,
    Cancelled
}
```

and a model property:

```csharp
[Column("status")]
public abstract OrderStatus Status { get; }
```

Explain that `--overwrite-types` controls whether simple existing C# types are replaced by database-inferred types.

## Implementation Plan

### 1. Add `ModelDirectory` Config

Add `ModelDirectory` to `ConfigFileDatabase`.

Add an effective `ModelDirectory` property to `DataLinqDatabaseConfig`.

Keep `DestinationDirectory` as an alias.

Normalize both paths before comparing conflicts. Use path normalization that is appropriate for local filesystem semantics, but do not require the directory to exist just to resolve config.

### 2. Resolve Alias and Warnings

Implement resolution:

- only `ModelDirectory`: use it
- only `DestinationDirectory`: use it and warn
- both same: use it
- both different: fail config or command resolution

Prefer returning a structured config/CLI failure over throwing.

### 3. Stop Using `SourceDirectories` in Active CLI Workflows

Update:

- `ModelGenerator`
- `SchemaValidator`
- `ModelReader`

so active workflows use `ModelDirectory` only.

`SqlGenerator` and `DatabaseCreator` already use destination output only; update names and aliases there.

### 4. Add `--fresh`

Add `Fresh` to `CreateModelsOptions`.

Map old `SkipSource` to the same effective behavior with a deprecation warning.

Rename internal options:

```csharp
ReadExistingModels = !options.Fresh && !options.SkipSource
```

Do not keep `ReadSourceModels` as the internal name. Names matter here because the goal is to kill the old mental model.

### 5. Read Existing Models From `ModelDirectory`

In `ModelGenerator.CreateModels(...)`, replace `SourceDirectories` parsing with:

```text
if ReadExistingModels and ModelDirectory exists:
    parse model files from ModelDirectory
else:
    skip existing model preservation
```

First-time generation should not warn just because the directory does not exist.

If the directory exists but contains no parseable database model, decide whether to warn or fail. My recommendation:

- empty directory: skip quietly or log a normal verbose line
- directory with C# files but no matching database model: fail unless `--fresh`

### 6. Keep and Rename Transformer Concepts

Keep `MetadataTransformer` initially, but rename surrounding comments and option names from source/destination to existing/fresh where practical.

A full transformer refactor can come later. This slice should change the product workflow without rewriting every merge algorithm.

### 7. Update Generated Header Wording

Coordinate with the generated-header plan.

CLI-generated model declaration files should link to the new docs page and mention supported edits.

Source-generator implementation files should keep a strict "do not edit implementation file" wording.

### 8. Update Docs

Update:

- `docs/Configuration files.md`
- `docs/getting-started/Configuration and Model Generation.md`
- `docs/CLI Documentation.md`
- any backend docs that mention source models for `create-models`
- dev plans that reference `SourceDirectories` if they would otherwise mislead future implementation

Do not claim the new workflow in user-facing docs before implementation lands.

### 9. Update Tests

Config tests:

- `ModelDirectory` is accepted
- `DestinationDirectory` still works as an alias
- both same path works
- both different paths fails clearly
- `SourceDirectories` presence is warned or ignored according to the chosen transition behavior

CLI/generator tests:

- first-time `generate models` works when `ModelDirectory` does not exist
- existing model files in `ModelDirectory` are parsed and preserve supported names/types
- `--fresh` ignores existing model files
- `--skip-source` is rejected by the new parser and help shows `--fresh`
- invalid existing model files fail unless `--fresh`

Validator tests:

- validation reads `ModelDirectory`
- validation no longer requires or reads `SourceDirectories`

Docs/header tests:

- CLI-generated model files contain the revised supported-edit header
- source-generator implementation files keep the stricter implementation-file header

## Compatibility and Migration Notes

This is a user-visible config migration.

Old:

```json
{
  "SourceDirectories": [ "Models/Source" ],
  "DestinationDirectory": "Models/Generated"
}
```

New:

```json
{
  "ModelDirectory": "Models/Generated"
}
```

Users who actually maintained a separate source model directory need a migration note:

1. Move any supported model declaration choices into the files under `ModelDirectory`, or regenerate once and edit the generated declarations there.
2. Move arbitrary custom code into partial classes outside `ModelDirectory` or into ordinary application files.
3. Remove `SourceDirectories`.
4. Replace any old `--skip-source` usage with `--fresh`.

This change may make some old workflows noisier for one release, but the long-term product shape is much cleaner.

## Non-Goals

- Do not remove `MetadataFromFileFactory`.
- Do not remove model-file parsing.
- Do not remove source-generator support for source-defined models.
- Do not remove `--overwrite-types`.
- Do not design a full explicit `ModelOverrides` config system in this slice.
- Do not guarantee arbitrary hand edits in generated files survive.
- Do not make generated files read-only.
- Do not keep `SourceDirectories` as a permanent supported workflow.

## Acceptance Criteria

- `ModelDirectory` is the documented config field for generated model files.
- `DestinationDirectory` remains a compatibility alias.
- Conflicting `ModelDirectory` and `DestinationDirectory` values fail loudly.
- `SourceDirectories` is no longer used by `generate models`, `validate`, or `diff`.
- `generate models` reads existing model files from `ModelDirectory` unless `--fresh` is used.
- `--fresh` ignores existing model files and generates from database metadata plus config.
- `--skip-source` is not kept as an alias in the new pre-1.0 CLI surface.
- Supported edits to class names, property names, relation names, and C# property types are preserved from existing model files in `ModelDirectory`.
- Generated CLI model headers link to documentation that explains supported edits, partials, and `--fresh`.
- User-facing docs explain that class and property names/types are intentionally editable, while arbitrary behavior belongs in partials.
